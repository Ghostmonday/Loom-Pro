# Deterministic Authority and LLM Containment

Loom uses a hybrid modeling approach that integrates mathematical analysis with language model classifications. Because language models operate probabilistically, Loom uses a strict authority hierarchy to ensure that LLM semantic inferences never undermine codebase security or structural integrity.

---

## 1. The Loom Authority Chain

To prevent non-deterministic behavior from affecting the repository structure, Loom enforces a strict downward chain of authority. Higher layers completely overrule lower layers:

```
Human Ratification
        ↓
Domain Rules (Layer 0)
        ↓
Deterministic Analysis
        ↓
LLM Semantic Inference
        ↓
Recommendations
        ↓
Presentation/UI
```

- **Human Ratification**: The ultimate authority. No code changes, structural merges, or GIV alterations can be committed without human validation and manual confirmation.
- **Domain Rules (Layer 0)**: Predefined security policies, directory constraints, and forbidden operations. These rules represent hard constraints that cannot be broken.
- **Deterministic Analysis**: Static AST compilation, Function-Realization Graph (FRG) generation, and Ollivier-Ricci curvature calculation. These calculations are mathematical guarantees and cannot be bypassed.
- **LLM Semantic Inference**: The probabilistic classification pass. It suggests semantic tags for files and methods (e.g. labeling a module as "auth" or "config").
- **Recommendations**: The semantic suggestions generated by the LLM. These recommendations are treated as *advisory only* and must pass both Deterministic Analysis and Domain Rules checks.
- **Presentation/UI**: The visual telemetry displayed on the Loom Blueprint Workbench.

---

## 2. Pipeline 3 (Semantic Synthesis) & Semantic Tagging

During Pipeline 3 (semantic synthesis), Loom clusters modules into functional domains. To classify ambiguous files (e.g. matching database helpers or network interfaces to appropriate work units), Loom uses a local LLM advisor to read the module context and return semantic tags.

Without containment, a misclassified tag could lead to a security breach. For example, if a database mutation helper is misclassified as a benign frontend config module:
1. The system might assign the module to a frontend worker.
2. The frontend worker's GIV allowed paths would include the database helper.
3. The worker could execute destructive mutations, bypassing the database write boundary.

---

## 3. LLM Containment and Mathematical Overrule

To prevent misclassification, Loom bounds all LLM semantic recommendations using two deterministic safeguards:

1. **Pre-flight Schema Locking**: Allowed and denied paths are locked into the mathematical graph before any agent is spawned. The LLM cannot mutate the static graph topology.
2. **Boundary and Curvature Contraction**: Curvature analysis determines physical coupling. If an LLM tag suggests grouping files across a detected **Dark Bridge** (κ < -0.3), the deterministic curvature engine overrides the LLM, forcing the coupled files into a single, isolated weld.

### Walkthrough: Misclassification Detection

When a mismatch between LLM suggestions and the deterministic topology occurs, Loom rejects the recommendation:

1. **Inference**: The LLM analyzes a database helper `db_cleanup.py` and incorrectly tags it as `benign_frontend_utility` due to name formatting.
2. **Deterministic Check**: The static analysis engine scans imports and detects that `db_cleanup.py` connects to the central data store via a negative curvature edge (κ = -0.52).
3. **Overrule**: Loom's graph engine overrides the LLM's recommendation, marks `db_cleanup.py` as a high-risk security-sensitive node, and enforces strict write isolation in the GIV, preventing any frontend workers from touching it.
4. **Audit Observability**: The Loom Workbench surfaces the node with a low confidence score, alerting the developer of the tag override.
